# Product Guidelines - Pixel Art Benchmark

## Design Philosophy
- **Objectivity First:** The application is a tool for evaluation. UI elements should be neutral and not distract from the assets being reviewed.
- **Precision:** Display pixel-perfect previews. Avoid anti-aliasing or scaling artifacts that could misrepresent the quality of the generated pixel art.
- **Transparency:** Clearly distinguish between model-generated content and human-provided reviews.

## User Experience (UX)
- **Reviewer Efficiency:** The review interface should minimize clicks and context switching. Use keyboard shortcuts for common actions (e.g., toggling grids, switching animation frames).
- **Responsive Previews:** Asset previews should load instantly and remain responsive during animation playback.
- **Read-Only Clarity:** In public mode, clearly indicate that the content is a static benchmark and disable all editing affordances.

## Visual Identity
- **Typography:** Use clean, monospaced or sans-serif fonts to reinforce a technical, diagnostic aesthetic.
- **Color Palette:** Use a high-contrast but muted palette (e.g., dark mode with clear semantic colors for scores and status).
- **Iconography:** Use standard icons for media controls (Play, Pause, Step) and technical toggles (Grid, Zoom).

## Content and Tone
- **Professional and Objective:** Use neutral language in benchmark reports. Avoid subjective or hyperbolic descriptions unless they are part of a user's specific qualitative review.
- **Technical Accuracy:** Use precise terminology when referring to sprite sheets (e.g., "3x3 grid," "frame-stepping," "alpha channel").
- **Consistency:** Ensure all model IDs and run identifiers follow the established naming convention consistently across the app.
